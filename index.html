Tornado-mongodb-example-app

simple example webserver using tornado-mongodb using nginx and supervisor.

View on GitHub Download .zip Download .tar.gz

Tornado and MongoDB needs no introduction. Here I am going to explain how to host an app/api with Tornado+MongoDB using nginx server as a proxy/load balancing and supervisor for process management.

This simple configuration can be done with few hours to up and run the site/api.

First let us see the Tornado simple code.

Template and Static file paths:

Tornado has option to define, templates and statics file paths. For static paths, ensure that, you're calling /static/ followed by file name. ie /static/static file (eg http://localhost/static/jquery.js). This can be changed using static_url_prefix option in the settings.

example code:

template_path = os.path.join( '/home/test/website/', "templates" ),
static_path = os.path.join( '/home/test/website/', 'assets' )

or simply,

template_path = '/home/test/website/templates',
static_path = '/home/test/website/assets'
Defining port/db settings

Passing the command line can be defined using "options" or can be parsed using parse_command_line option.

define( "port", default = 8888, type = int )
parse_command_line()
Defining handlers for different paths

It is easy to map which url path should be served by which function. adding that in handlers takes care of that. It is also possible to map based on regular expressions too.

handlers = [( r"/", HomeHandler ),
            ( r"/api", MyAPIHandler ),
            ( r"/demo", DemoUI )
            ]

functions

class DemoUI( BaseHandler )
    def get( self ):
        number= [i for i in range(0,10) ]
        self.render( "one.html" , title = "one",numberlist=number )
class HomeHandler( BaseHandler ):
    def get( self ):
        self.render( "index.html" )

** render** renders the templete with the variables.

Setting multiple inheritance

Multiple inheritance useful for things such as db connections etc. DB connections can be resued using this.

super( Example, self ).__init__( handlers, **settings )
BaseHandler for property definition

once the property is defined in the basehandler, it will be availble to all the handlers. In the example, property for using db connection is defined.

create connection in the init function

session = MongoClient()
self.firstCol = session.firstDB.firstCol
self.secondCol = session.secondDB.secondCol

define the property in a basehandler calss

class BaseHandler( tornado.web.RequestHandler ):
    @property
    def secondCol( self ):
        return self.application.secondCol
    @property
    def firstCol( self ):
        return self.application.firstCol

Now, calling self.firstcol from any function refers to the connections

Logging

tornado enables default logging into access,error,general loggers. This can be turned off using --logging=none option when starting the server. It is also easy to send the log to different file. But issue come when you try to turn off default logging and log only needed. For the default logger should be propageted false. In the example only access log is turned off. Also rotating file handler for logger using to roate the log files if maximum size is reached. 

options.logging = None
access = logging.NullHandler()
access.setLevel( logging.DEBUG )
logging.getLogger( "tornado.access" ).addHandler( access )
logging.getLogger( "tornado.access" ).propagate = False
LOG_FILENAME = '/home/test/logging.log'
handler = logging.handlers.RotatingFileHandler( LOG_FILENAME, maxBytes = 1 * 1024 * 1024, backupCount = 10 )
formatter = logging.Formatter(
        '%(asctime)s process [%(process)d]: %(message)s',
        '%b %d %H:%M:%S' )
handler.setFormatter( formatter )
logger = logging.getLogger()
logger.addHandler( handler )
logger.setLevel( logging.DEBUG )
using xsrf cookie to make app secure

it is easy to restrict access to the app only via browser thus restricting crawling options. add xsrf_cookies = True, to the settings and add this to the html form

{ % module xsrf_form_html() % }

when loading the form, xsrf cooking will be rendered as hidden parameter.

if there is a need to disable xsrf for only one handler and enable for others, add this to the handler function. This will remove the check for that handler only.

   def check_xsrf_cookie( self ):
        pass

Confiuring nginx

Tornado is an web server itself, so why we need Nginx as a additional component?

nginx is used as a proxy server and load balancer. nginx sits between the server and client, distributes the traffic and offers additional things as rate limiting etc.

Load Balancing with nginx is easy. Let us say, you have four process running in a single machine and you want each process to handle the requests from users simultaneously you can configure that in nginx conf.

Each process running separate port will be connected to ngnix sever and nginx’s single port shown to the user.

defining app ip settings

upstream myapp
    {
        server 127.0.0.1:8888;
        server 127.0.0.1:8889;
        server 127.0.0.1:8890;
        server 127.0.0.1:8891;
    }

here, one server is going to listen all the four ips. Defining various ips using various ports is also available.

Next defining server option

   server
    {
      listen 127.0.0.1:80;

      location /
      {
        proxy_pass http://myapp;
        proxy_set_header X-Real-IP $remote_addr;
        limit_req zone=index burst=1;
      }

      location /api/
      {
        proxy_pass http://myapp;
        proxy_set_header X-Real-IP $remote_addr;
        limit_req zone=api burst=10 nodelay;
      }

    }

listen is self explonatory. It exposes nginx running port to public.

** proxy_pass** takes care of the load balancing between requests. Without that load wont be distributed.

** proxy_set_header** would set the actual ip of the request in the request headers. Which means, requests has the actual ip from which the app is accessed. If the app is exposed to public, the actual ip from which customers accessing would be forwared to tornado process.

** limit_req zone** option defines the number of requests the server can handle per second or mintue. that is defined in this line

limit_req_zone $binary_remote_addr zone=api:1m  rate=100r/s;

** burst** allows how many requests can be handled at a time and nodelay configures if the requests should be put into the queue or not.

if rate is 100r/s and burst is 10 with nodelay, then 10 requests would be processed in batches.

Using Supervisor

Handling start/shutdown of the processes can be handled using Supervisor

let us say, four processes needs to be started for tornado, use this supervisor configuration to start those processes

[program:firstapp]
process_name=firstapp%(process_num)s
directory=/home/test/
command=python /home/test/tornado_example.py --port=%(process_num)s --logging=debug
stdout_logfile=/home/test/out-%(process_num)s.log
stderr_logfile=/home/test/err-%(process_num)s.log
startsecs=2
numprocs=4
numprocs_start=8888

Supervisor can restart the processes if there is an expection or system reboot.
